Build your first API with Mezzio

Mezzio is PHP's PSR-15 middleware microframework. This guide takes you from an empty directory to a working JSON API — explaining every concept you pass through on the way, and nothing you don't need yet.

PHP 8.1+ No Mezzio experience assumed Read top to bottom · ~60 min You finish with a running API

01Introduction — what Mezzio is

Mezzio (formerly Zend Expressive, now part of the Laminas Project) is a microframework for building PHP applications out of middleware. Instead of giving you a big framework with controllers, an ORM, a templating layer, and a queue system baked in, Mezzio gives you one sharp idea: an HTTP request flows through a pipeline of small, composable layers, and one of them produces the response.

Mezzio is built on open PHP standards rather than framework-specific conventions. Two of them matter for this guide, and we'll introduce each properly when it comes up:

  • PSR-15 — the standard interface for HTTP middleware and request handlers (sections 06 and 09).
  • PSR-11 — the standard interface for dependency injection containers (section 10).

Because everything is standards-based, the parts are swappable: Mezzio lets you pick your router, your DI container, and your template engine at install time — or replace them later without rewriting your app.

Micro-framework vs. full-stack framework

If you've used Laravel or Symfony, here's where Mezzio sits:

Full-stack (Laravel, Symfony)Micro (Mezzio)
Ships with ORM, auth, queues, mail, templating, CLI — batteries included An HTTP pipeline, a router, a DI container. You add the rest as you need it
Structure Convention-driven: the framework decides where things live and how they're wired Explicit: you wire every route, middleware, and service yourself — nothing happens by magic
Learning curve Fast start, large surface area to learn eventually A handful of concepts (this guide covers all of them), then you're done
Sweet spot Product apps that want everything integrated APIs, microservices, and apps that want a small, explicit, standards-based core

When to reach for Mezzio

  • You're building a JSON API or microservice and don't want to carry a full-stack framework for it.
  • You want explicit wiring you can read — every route and middleware in your app is listed in two files you own.
  • You care about PHP standards (PSR-7/11/15) so your code and third-party packages compose cleanly.
  • You want to choose your own components — swap FastRoute for another router, or laminas-servicemanager for another container.

If instead you want an admin panel, an ORM, and auth scaffolding out of the box next week, a full-stack framework will serve you better. Mezzio optimizes for small, sharp, long-lived services.

The key files, one by one

public/index.php — the entry point

Every single HTTP request lands here; there are no other entry scripts. It builds the container, pulls the Application out of it, loads the pipeline and routes, and runs the app:

config/pipeline.php vs config/routes.php — the two files you'll live in

These two files look similar (both return a function that receives $app) but do different jobs, and the difference matters:

  • pipeline.php answers: "what happens to every request?" It registers middleware — error handling, routing, dispatch — in the exact order requests pass through them. It never mentions URLs like /api/products.
  • routes.php answers: "which handler serves this particular URL + method?" It maps GET /api/ping to PingHandler. It says nothing about cross-cutting concerns like error handling.

Rule of thumb: adding an endpoint? Touch routes.php. Adding behaviour that applies across endpoints (auth, CORS, JSON body parsing)? Touch pipeline.php.

config/config.php + config/autoload/ — configuration aggregation

config.php collects configuration from every module's ConfigProvider and every file in config/autoload/, and merges it all into one big array. Files ending in .global.php are shared settings you commit; .local.php files are machine-specific secrets that are git-ignored and win over global. Section 11 unpacks the merge rules.

src/App/ — your module

Your actual application code lives here. Two patterns to notice:

  • Handler/ — one class per endpoint (or resource). A handler receives the request and returns the response; it's the "controller" of the Mezzio world, covered in section 09.
  • ConfigProvider.php — the module's manifest. It tells the container "here are my services and how to build them". Covered in section 10.

04The request lifecycle — how data flows

This is the single most important section of the guide. Every other concept — routing, piping, handlers — is one stop on the journey a request makes through your app. Learn the journey once and everything else has a place to hang.

What

The request lifecycle is the path every HTTP request takes: it enters at public/index.php, is converted into a PSR-7 request object, travels inward through each piped middleware in order, gets matched to a route and dispatched to a handler, and the handler's response travels back outward through the same middleware in reverse before being emitted to the client.

Why

Frameworks with controllers and hooks scatter "what happens to a request" across many places. Mezzio makes the whole journey one explicit, ordered list (pipeline.php). When you wonder "where do I add auth?" or "why did this return 404?", you answer it by walking the pipeline in your head — there is nowhere else to look.

How

Trace one request — GET /api/ping — through the skeleton you just installed:

A request goes in through each layer, reaches the handler, and the response comes back out through the same layers in reverse.

In prose, the numbered steps are:

  1. The web server hands the raw request to public/index.php.
  2. index.php builds the container, gets the Application, and registers everything in pipeline.php and routes.php.
  3. $app->run() turns PHP's superglobals into a PSR-7 ServerRequest object — from here on, "the request" is a value you pass around, not global state.
  4. The request enters the pipeline and passes through each middleware in the order they were piped.
  5. RouteMiddleware asks the router "which route matches GET /api/ping?" and attaches the answer to the request.
  6. DispatchMiddleware reads that answer and invokes PingHandler.
  7. The handler returns a PSR-7 Response, which travels back out through the same middleware in reverse order — each gets a chance to inspect or modify it.
  8. An emitter writes the status line, headers, and body to the client.
Mental model

Airport security, both directions. Your request is a traveller. It enters the terminal (index.php), then passes a fixed sequence of checkpoints in order — ticket check (error handler armed), passport control (routing: which gate is this traveller even going to?), security screening (any middleware you add) — until it reaches its gate (the handler), where the actual business happens. The response is the return trip: it walks back past every checkpoint in reverse, and any checkpoint can stamp it on the way out (add a header, log the timing). And crucially: a checkpoint can turn a traveller away early — no boarding pass, no gate. That's a middleware short-circuiting with a 401.

Use case

Your API suddenly returns 404 for a route you swear you registered. With the lifecycle in your head, you debug in order: did the request reach RouteMiddleware (is your route in routes.php, spelled right, with the right method)? Did something before routing short-circuit it? Is DispatchMiddleware piped after RouteMiddleware? Nine times out of ten it's a typo in routes.php — and now you know exactly which of the two files to open.

Keep this diagram handy

The rest of this guide keeps referring back to this picture. Routing (§05) is what RouteMiddleware does. Piping (§07) is how the checkpoints get installed. Stacking (§08) is the in-and-back-out nesting. Handlers (§09) are the gate.

05Routing

What

Routing maps an HTTP method + URL path to the handler that should serve it. In Mezzio, all routes are declared in config/routes.php using methods on $app.

Why

Without a router, index.php would be a giant if/else on $_SERVER['REQUEST_URI']. The router centralizes that mapping, handles dynamic segments (/products/42), enforces HTTP methods, and — because it runs as middleware (RouteMiddleware, step 5 in the lifecycle) — the match result is attached to the request where later layers can read it.

  • Route names (the third argument) let you generate URLs later with the UrlHelper instead of hard-coding paths.
  • Regex constraints like {productId:\d+} reject garbage early — /api/products/abc never reaches your handler.
Use case

You're adding "fetch one product" to an existing products API. You add a single line — $app->get('/api/products/{productId:\d+}', ProductHandler::class, 'products.get') — and write the handler. No pipeline changes, no config changes: endpoints live in routes.php, full stop.

06Middleware

What

Middleware is a class that sits in the path of the request. It receives the request plus a $handler representing "the rest of the pipeline", and it must return a response — either by delegating onward ($handler->handle($request)) or by producing one itself. The contract is PSR-15's MiddlewareInterface:

Why

Cross-cutting concerns — authentication, logging, CORS, body parsing — apply to many endpoints. Copy-pasting them into every handler doesn't scale, and base-class inheritance couples everything together. Middleware solves this by making each concern a self-contained layer you compose into the path. It's kept separate from handlers on purpose: a middleware is infrastructure the request passes through; a handler is the business logic it was heading to.

Mental model

Middleware is a layer you pass through; a handler is the destination you arrive at. On a road trip, toll booths, border checks, and weigh stations are middleware — you pass through many of them, in order, and any one can turn you back. The city you're driving to is the handler — there's exactly one per trip, and reaching it is the whole point. Nobody confuses a toll booth with a destination; keep the same distinction between middleware and handlers.

Use case

Product asks: "log every API call with its response time." That's not any single endpoint's job — it's a TimingMiddleware: record hrtime(), call $handler->handle($request), log the elapsed time, return the response. Pipe it once (next section) and every current and future endpoint is covered.

07Piping

What

"Piping" is Mezzio's word for registering middleware onto the application's pipeline with $app->pipe(). The order of pipe() calls in config/pipeline.php is the order requests flow through the middleware.

Why

Something has to decide which middleware run, and in what sequence — should errors be caught before or after auth? Should the body be parsed before routing? Rather than hiding that ordering in annotations or framework internals, Mezzio makes it one readable file: the pipeline you pipe is the pipeline you get.

How

There are three scopes at which middleware can attach, from broadest to narrowest:

How to choose: pipe globally for true infrastructure (error handling, logging), by path for concerns that cover a section of your API (auth on everything under /api, but not the public health check), and per-route for one endpoint's specific needs (validating one form's payload). Piped middleware runs for every matching request even if no route matches; route-attached middleware runs only after routing has already succeeded for that route.

Mental model

Literal plumbing. Each pipe() call fits one more segment of pipe onto the line, and the request is the water: it enters at the first segment fitted and flows through every segment in the order they were joined. A path-segregated pipe is a branch line that only carries water headed for one wing of the building. And just like real plumbing, order is physical: fitting the water meter after the outlet means the outlet's usage never gets counted — pipe your ErrorHandler first or errors upstream of it go uncaught.

Use case

Your API needs to accept JSON request bodies. PSR-7 doesn't parse JSON automatically, so you pipe BodyParamsMiddleware (from mezzio/mezzio-helpers) once, before routing — and from then on every handler can call $request->getParsedBody() and get an array, whether the client sent JSON or a form. One pipe() line, app-wide capability. We do exactly this in the worked example (§12).

08Middleware stacking

What

Stacking is what multiple piped middleware become at runtime: a nested chain where each layer wraps everything piped after it. Middleware #1 receives the request first and returns the response last; the innermost layer (ultimately your handler) runs in the middle.

Why

The wrap-around shape is the whole trick: because each layer gets control both before and after everything inside it, one middleware can time a request (before + after), catch exceptions thrown anywhere deeper (that's exactly why ErrorHandler is piped first), or veto the request before deeper layers waste any work.

How

Delegation is explicit: a middleware continues the chain by calling $handler->handle($request), and whatever that returns is the response from every layer inside it.

Short-circuiting: not calling $handler->handle()

Delegating is a choice. If a middleware returns a response without calling $handler->handle($request), every deeper layer — including routing and your handler — simply never runs. That's not a failure mode; it's a feature you'll use deliberately:

  • ApiKeyMiddleware returns 401 → the handler never executes for unauthenticated requests.
  • A rate limiter returns 429 → no wasted database work.
  • A cache middleware returns a stored response → the whole inner stack is skipped on a hit.

The bug to avoid is short-circuiting by accident: if your middleware forgets to call $handler->handle($request) (or drops its return value), every request stops at that layer. If every route suddenly returns the same response or a blank 200, look for the middleware that forgot to delegate.

Passing data down the stack: request attributes

Layers don't just wave the request through — they often need to hand results to the layers inside them: "I already validated this payload", "the authenticated user is #42". The channel for that is the request itself, via PSR-7 attributes: an outer middleware attaches data with $request->withAttribute('key', $value), passes the new request inward, and any deeper middleware or handler reads it back with $request->getAttribute('key'). You've already seen this mechanism without naming it — it's exactly how the router hands {productId} to your handler in section 05.

Two habits keep attributes sane. First, attributes travel inward only — a deeper layer can't send data back out through the request; anything for the return trip goes on the response (a header, usually). Mental model

Russian nesting dolls. Piping assembles a matryoshka: the first-piped middleware is the outermost doll, the handler is the solid doll at the very center. Opening the dolls is the request travelling inward; closing them back up, innermost first, is the response travelling out — so the outermost doll is first to see the request and last to touch the response. Short-circuiting is deciding not to open the next doll: whatever is inside it stays untouched.

Use case

You add caching to a slow reports endpoint. A CacheMiddleware piped at /api/reports checks Redis first: on a hit it returns the cached response (short-circuit — handler and database never run); on a miss it delegates inward, then stores the response it gets back on the way out. Both behaviours in one small class, possible only because of the wrap-around stacking shape.

09Request handlers

What

A request handler is the class at the center of the pipeline — the destination. It implements PSR-15's RequestHandlerInterface: one method, handle(), that takes the request and must return a response. Unlike middleware, it receives no $handler argument — there is nothing further to delegate to.

Why

The pipeline needs a guaranteed endpoint — something that always produces a response so every middleware above it has a response to pass back out. Handlers are also where Mezzio replaces the "controller with many action methods" pattern: one handler class per endpoint keeps each class small, single-purpose, and trivially testable (it's just an object with one method — no framework booting required).

10Dependency injection & the service container

What

The service container is a registry that knows how to build every object ("service") your app needs — handlers, middleware, database connections — and hands out ready-made instances on request. Mezzio talks to it through PSR-11, a two-method interface: $container->get($name) and $container->has($name). The skeleton's implementation is laminas-servicemanager.

Why

Your handlers need things: a repository, a PDO connection, an API client. Constructing those inside the handler (new PDO(...) in handle()) hard-codes the dependency, makes testing painful, and scatters connection details everywhere. Dependency injection flips it: the handler declares what it needs in its constructor, and the container assembles it. You saw the container born on line one of the lifecycle — $container = require 'config/container.php' — and in fact every handler and middleware in your app is pulled from it; that's why you register class names in routes.php rather than instances.

How

Three pieces work together: a service with constructor dependencies, a factory that builds it, and a ConfigProvider entry that registers the factory.

Less boilerplate, when you want it

Writing a factory per service is the explicit default, but laminas-servicemanager also ships ReflectionBasedAbstractFactory, which inspects a constructor and resolves type-hinted dependencies automatically — register it as the factory for a service and skip writing one by hand (Handler\ProductListHandler::class => ReflectionBasedAbstractFactory::class). Handy in development; explicit factories are cheaper at runtime and easier to debug.

Mental model

A vending machine for objects. You press a button labelled with a service name ($container->get(ProductRepository::class)) and a fully-assembled instance drops out. You don't see — or care — how it was put together. The factories are the restocking instructions taped inside the machine, and each module's ConfigProvider is the sheet telling the machine which instructions go with which button. One more property: it's a vending machine with a memory — ask for the same service twice and you get the same instance back, not a fresh one.

Use case

Swap the in-memory ProductRepository for a PDO-backed one (§13). Handlers don't change at all — they just declare they need a repository. You change one factory to build the PDO version, and every consumer gets it. That isolation is what the container buys you.

11Configuration

What

Mezzio configuration is one big PHP array, assembled at startup by the ConfigAggregator in config/config.php. It merges, in order: each module's ConfigProvider, then every file in config/autoload/ matching *.global.php, then every *.local.php. The merged array is itself available from the container as the service named 'config'.

Why

Different settings have different owners: a module ships defaults, your project sets shared values, and each machine (your laptop, CI, production) has its own secrets. One file can't serve all three. The aggregator lets each layer contribute a fragment, with a simple rule — later wins — deciding conflicts. That's also why *.local.php files are git-ignored: secrets stay on the machine they belong to, and they override whatever the committed files say.

How

Mental model

Layered transparency sheets. Picture each config source as a transparent sheet with some cells filled in, stacked on an overhead projector in aggregation order: framework defaults on the bottom, your module's sheet next, *.global.php sheets above that, *.local.php sheets on top. Looking down through the stack, you see the topmost value written in each cell — everything underneath still exists, it's just overridden. Adding a machine-specific tweak is laying one more small sheet on top, never editing the sheets below.

Use case

Your app needs a Stripe key. You commit stripe.global.php with 'secret_key' => '' (documents the shape, safe placeholder), and on each environment create stripe.local.php with the real key for that environment. Production and your laptop run identical code with different top sheets — no if ($env === 'prod') anywhere. Section 15 builds this exact setup.

Config caching gotcha

When ConfigAggregator::ENABLE_CACHE is on (production mode), the merged array is written to data/cache/config-cache.php and config files are never re-read. If a config change mysteriously doesn't apply, run php bin/clear-config-cache.php. In development mode (composer development-enable) caching is off and this never bites.

An entity is a plain PHP class whose attributes declare, field by field, how a table row maps onto it. Here is the products table from §13 as an entity:

Mental model

A column-header legend taped to a spreadsheet. A table row is a line in a spreadsheet; an object is the real thing it describes. The entity's attributes are the legend taped across the top: #[ORM\Column(length: 120)] above $name means "whatever sits in the name column belongs in this property." The EntityManager is the clerk who reads the legend fluently in both directions — SELECTed rows come back to you as Product objects (Doctrine calls this hydration), and the changes you make to those objects get typed back up as INSERT/UPDATE statements. You never write the translation; you only keep the legend accurate.

15Third-party services — same pattern, every time

Here's the payoff of learning the factory triangle once: every external service joins your app the exact same way. An email provider and a payment provider look wildly different in their docs — in Mezzio they're the same five moves:

  1. Put credentials in config/autoload/*.local.php (git-ignored).
  2. Write a small client class of your own that wraps the vendor SDK behind methods named after what your app does ("send an order confirmation", "take a payment").
  3. Write a factory that reads config and assembles that client.
  4. Register it in the ConfigProvider.
  5. Inject it into whichever handler needs it.

Move 2 is the one people skip — don't. If handlers use StripeClient directly, Stripe's API is smeared across your codebase; behind your own client class, swapping providers (or faking payments in tests) means changing one file.

You've gone from zero to a Mezzio API with routing, middleware, DI, a database, third-party services, and API-shaped error handling. Where to go from here

  • Official Mezzio documentation — the reference for everything this guide compressed; the "Cookbook" section is particularly practical.
  • Container deep dive — delegators, lazy services, and the other laminas-servicemanager tools beyond factories and invokables.
  • Modular structure — when the app outgrows one src/App module, split it into modules (src/User, src/Billing…), each with its own ConfigProvider. You chose "flat" at install; "modular" is the same ideas, multiplied.
  • mezzio-authentication — session, basic-auth, and OAuth2 adapters, delivered (of course) as pipeline middleware.
  • mezzio-swoole — run the same app on Swoole/OpenSwoole as a long-lived async server instead of PHP-FPM; big throughput wins, zero application rewrites.
  • Event dispatch (PSR-14) — decouple side-effects ("order created → send email") from handlers using an event dispatcher.
  • Templating — if this API grows HTML pages, add Twig or laminas-view; rendered templates are just another way for a handler to build its response body.
